# 2026-07 — Phase 20c: the drink that isn't one number

*Taiwan runs on hand-shaken tea, and "可不可熟成紅茶一分糖少冰加珍珠" is not a product — it's a base drink times a sugar level times ice times toppings times cup size. Logging it as a single LLM-guessed calorie count throws away the one thing a fitness app should care about: the sugar. So a drink stops being a row and becomes an equation.*

## The problem

A convenience-store onigiri (Phase 20b) is a fixed product: one barcode, one label, one set of numbers. A hand-shaken drink is the opposite. The same 熟成紅茶 at 全糖 versus 無糖 differs by hundreds of calories; a scoop of pearls adds ~120 more. The customization *is* the nutrition — and it's exactly what a single "bubble tea ≈ 300 kcal" estimate erases.

Worse, the data barely exists. Convenience stores publish per-item labels; hand-shaken chains mostly publish, at best, the full-sugar calories of a drink and nothing per sugar level. So the catalog approach that worked for onigiri — store the finished item, hit it exactly — can't work here. There is no finished item to store. There are only thousands of combinations of a few dozen bases.

## Decisions

**A drink is stored as an unsweetened base plus a scalable sugar component, not as a finished row.** Each base carries its 無糖 calories (for a milk tea, that's the milk; for plain tea, near zero) and a separate `full_sugar_g` — the grams of syrup at 全糖. Calories are computed at log time: `base + full_sugar_g × sugar_pct × 4 + toppings`. Because most chains only publish full-sugar figures, the base is *derived*: `unsweetened = full_sugar_calories − full_sugar_g × 4`. The payoff is that 一分糖 (10%) and 全糖 are now genuinely different numbers instead of the same guess, and sugar becomes a first-class column on every logged item — the substrate for the "you're over your sugar this week" coaching planned later. The standard Taiwan sugar ladder (全糖 1.0, 少糖 0.7, 半糖 0.5, 微糖 0.3, 一分 0.1, 無糖 0) lives as data, so a shop with an extra rung like 微微糖 (≈0.15) is one row, not a code change.

**The LLM extracts structure; deterministic code does everything else.** The model's only job is to pull the raw surface terms out of the sentence — brand, base, sugar word, ice, toppings — verbatim, no translation, no math. A deterministic resolver then maps those surface terms onto canonical concepts through a curated per-brand dictionary: 可不可's 白玉 *is* pearls; one shop's 微微糖 *is* 15%. A deterministic engine computes the calories. This keeps the expensive, non-reproducible part (understanding a messy sentence) with the LLM and the part that must be auditable and correctable (what a topping costs, what a sugar word means) in reviewed data. A wrong mapping is a one-line dictionary edit and a re-run, not a prompt-tuning session — and the model never gets to hallucinate that 白玉 is tapioca one day and something else the next.

**The engine is an accelerator, never a gate.** The beverage path sits between the personal-food lookup and the generic catalog in the estimate chain, and it only activates when the message actually parsed as a hand-shaken drink. Every miss — a brand not yet curated, a base not in the catalog, a sugar word or topping the dictionary doesn't know — falls straight through to the existing estimator. Unmapped components are flagged for review rather than guessed. The guarantee is that turning this on can only make an estimate better or leave it unchanged; it can never error, and it can never be worse than the LLM number we had before.

**Honesty is encoded in the data, and the whole method is a reusable skill.** Hand-shaken nutrition is thin, so every base is marked `curated_estimate` with its source, and where a chain doesn't publish protein/fat/carbs I estimate them from the drink's composition — reconciled so the macros add back up to the calories — rather than writing a confident zero, which on a fresh-milk latte is just visibly wrong. The decomposition is what makes this defensible without per-cup official data: the sugar arithmetic is exact, toppings are standard, and each error source is isolated and separately correctable instead of buried in one opaque guess. As with the convenience-store catalog, the entire curation procedure — the derivation, the brand-name rule, the red lines — is written up as an agent skill so the next shop is a repeatable exercise, not a rediscovery.

## Hindsight, honestly

- **A lookup keyed on the wrong string is worse than no lookup — it fails silently.** The first cut stored brands under an English slug while the model emits the Chinese name, so the exact-match lookup could never hit and the entire feature was quietly inert. Everything compiled, every test with a hand-built store passed, and nothing worked end to end. The fix was trivial (store the name the producer actually emits); noticing was the hard part. Any key that crosses a boundary should be tested against a value the *other* side really produces, not one the test author typed.
- **Refusing to write zero is a data-quality decision, not a nicety.** Padding unknown macros with 0 would have shipped a milk drink with zero protein and zero fat — a number a user can see is wrong at a glance, which poisons trust in every other number too. Estimating from composition and labeling it an estimate is more honest than a precise-looking zero.
- **Model the thing that varies.** The temptation was to let the LLM return one calorie figure per drink and move on. It would have been a fraction of the work and structurally incapable of representing sugar level — the single most important variable. Choosing the harder data model was choosing to be able to answer the question the product exists to answer.
