# 2026-07 — Known-food passthrough: the cheapest LLM call is the one you never make

*Every typed meal used to cost a model call. But "御飯糰 一個" and "牛肉麵 一碗" are the same handful of foods logged over and over — paying an LLM to re-read them is paying to solve a problem you've already solved. So the model stops being the first responder and becomes the fallback: known foods resolve to a log with zero tokens, and only genuinely new input reaches the model.*

## The problem

FitNeko's text pipeline already had a deterministic front line — ordered intent rules that catch summaries, confirmations, and deletions with no model call. But anything that looked like *food* fell through to one combined LLM call that classified, parsed, and estimated in a single round-trip. That call is the product's per-message cost driver, and it fired even when the message was a food the user (or the catalog) had seen a hundred times.

The obvious move — "cache the answer" — is deceptively hard for natural language. "牛奶 500ml", "500 毫升牛奶", "喝了0.5公升鮮奶" are one fact spread across three surfaces, and a cache keyed on raw text hits almost never. And the dangerous failure mode isn't a miss — it's a *wrong* hit: shaving a message down to a food name and confidently logging the wrong thing. A cost optimization that occasionally records the wrong food is not an optimization.

## Decisions

**A zero-token resolver sits between the intent rules and the LLM, gated by a known-food whitelist.** When the intent rules don't match, a resolver tries to reduce the message to a single food + quantity + unit and match it against *known* sources. The key safety inversion: the extractor is untrusted. A candidate only fires if it resolves against a known source, so a mis-extraction (a negation, a compound name like 牛奶糖 vs 牛奶, an unknown unit) finds nothing and *abstains* — falling through to the LLM, which costs a call but never logs a wrong food. The whitelist, not the extractor, is the source of truth. The trade-off is extra database lookups on the hot path, but these are indexed point-lookups measured in single-digit milliseconds against a fallback measured in seconds; a miss adds noise to a call that was already going to be slow.

**Nutrition is cached per unit, with units canonicalized to a base.** The cache key drops quantity entirely: values are stored per one unit and scaled at read time, so every quantity of the same food shares one entry. Units are canonicalized before keying — 毫升/ml/cc collapse to `ml`, 公升 converts into that base, the count words settle onto one spelling — so "500ml", "500 毫升", and "0.5公升" all land on the same entry and reuse each other's estimate. The subtlety that bit later: once units can be spelled at different scales, the scaling arithmetic has to run on the *base* quantities, not the raw numbers, or 0.5 公升 read back against 250 毫升 is wrong by a factor of a thousand.

**Per-user sources rank above the shared ones, because your own numbers beat a generic guess.** The resolver checks, in order: your saved food library, your own past *authoritative* logs, the shared Taiwan catalog, then prior cached estimates. The per-user layers come first deliberately — a food you defined, or a correction you made by hand, is more true for you than any generic row. Only genuinely authoritative log rows qualify as a source (the numbers you provided or a scanned label), never the app's own past AI guesses; re-serving those would add nothing the estimate cache doesn't already do, and would let one stray value echo forever. Most-recent-wins, so a later correction quietly supersedes an earlier one.

**Per-user values must never leak into the shared cache.** This one surfaced in review, not design. The estimator already refused to seed its cross-user estimate cache with nutrition a user typed inline — a deliberate "don't let one person's number become everyone's number" rule. But the new per-user resolver sources produced values that slid past that guard, so a user's custom "沙拉 = 800 kcal" could quietly become the generic answer every other user reads — and cache hits are sticky, so a bad value doesn't get re-estimated, it gets served. The fix was to make provenance the gate: values sourced from a person (their library, their logs) serve only that person and are barred from the shared cache. The generic cache stays generic.

## Hindsight, honestly

- **The cache turns spend into capital.** Every LLM call I pay for isn't only a cost — it writes its answer back, so the more the model is called, the more often the *next* message resolves for free. Cost per log falls as usage rises instead of scaling with it, and the early bill is really an investment in a growing corpus of known foods that pays back on every repeat. In a food domain — where a small set of items gets logged over and over — that compounding is the entire economic case for the fast path: you spend now to stop spending later.
