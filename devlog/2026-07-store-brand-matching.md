# 2026-07 — Store brands: when naming the shop hid the food

*The catalog already had the item. A user typing `全家一配筋子銀鮭飯糰` should have been the easy case — a real product, in the database, with official numbers. Instead it was the case that missed, because the one extra word that made the input more specific was the word that made it match nothing. The fix was to teach every match site the same trick: pull the brand off, but keep it.*

## The problem

Food matching against the shared catalog — and against the per-food estimate cache behind it — was exact string match on the food name. That works until brands enter the text. Convenience-store rows are stored brand-free: the catalog holds `一配筋子銀鮭飯糰` with the brand as a separate column, not baked into the name. So `全家一配筋子銀鮭飯糰` compared unequal to the very row that described it, fell through, and got sent to the LLM for a guess — paying to estimate a food the database already knew exactly. The estimate cache had the identical blind spot: keyed on the raw surface, a brand-prefixed repeat never reused the work done for its brand-free twin.

The irony is that the *more* informative input matched *worse*. And the pipeline already knew how to do this right for one category — hand-shaken drinks parse brand into its own field before matching — but regular foods carried the brand inline, where it could only get in the way.

## Decisions

**One brand parser feeds every match site, so the keys line up.** A single function turns a food surface into `(brand token, clean name)` using a curated list of store surfaces, and it is the *only* place brand logic lives. It is called at three points that must agree: the catalog lookup, the cache read, and the cache write. The reason they must share it is the whole game — if the model parses the brand out on the way in but the cache wrote its answer under a differently-shaped key on the way out, the next identical message misses and pays for the model again. Aligning the three keys is what turns "parsed once" into "free forever after."

**Try the whole string first, strip the brand only as a fallback.** Stripping is dangerous for brands that are also part of a product's identity — `統一肉燥麵` is not a generic `肉燥麵`, and blindly removing the leading `統一` would log the wrong food's numbers. So the catalog is queried with the literal input first; only if that misses does the brand-stripped name get a turn, and among same-name rows the matching brand sorts ahead of the generic one. The trade-off is up to two lookups on a miss — cheap indexed point-reads against a fallback that was already going to call an LLM.

**Brand stays in the cache key rather than being stripped away.** The estimate cache is the generic-fallback tier; the catalog is the brand-accurate source. Keeping the brand token in the cache key means `全家關東煮` and `7-11關東煮` cache separately instead of one silently answering for the other, and the catalog stays the place where brand precision actually lives. The accepted cost: the key format changed, so existing cached estimates re-warm once on their next miss — a one-time cold, not lost data.

**The model learns to separate brand from name, mirroring the drink pipeline.** Rather than lean entirely on string-stripping — which can't reliably tell a brand prefix from a name that merely starts with one — each parsed food item now carries the brand as its own field, the same shape the drink extractor already used. But the shared parser still strips any brand the model left inline, so the two mechanisms back each other up: structured extraction handles the ambiguous cases, string-stripping catches what slips through. Defense in depth, not either-or.

**A divergence surfaced in review, not design.** The zero-token resolver on the hot path had quietly simplified itself into stripping the brand *before* every catalog query — dropping the "whole string first" guard that the slower service path still honored. Two code paths meant to do the same thing had drifted, and no test told them apart, because on today's brand-free catalog rows both happen to return the same answer. The gap only bites once curated products start carrying a brand in their stored name — exactly the future this work is meant to enable. The fix restored the guard on both paths and added a test that fails on the strip-first behavior, so the guarantee can't silently regress again.

## Hindsight, honestly

- **Writing this up is how I learned half the phase hadn't paid off yet.** The string parser carries the actual matching — on the zero-token hot path, which never calls the model, and as the backstop everywhere else — and because the structured LLM brand extraction resolves against the *same* curated brand list, it barely moves the match rate on its own. Its real justification was persisting a clean brand per logged item, for future per-brand curation and display. Auditing the code for this entry, I found that persistence was never wired: the log row's brand column is still default-empty, and the extracted brand only feeds the cache key. So the LLM half shipped rolled-out-but-not-harvested. Next time I'd either defer it until the consumer that reads it exists — plain YAGNI — or wire that consumer in the same phase, so the field earns its place the day it lands instead of waiting for one.

- **The brand-token ↔ catalog-brand contract lives in a comment, and that's the landmine.** Brand-priority ordering only works when a parser's token equals the exact brand value the catalog importer wrote — and today only one store's data is real, so every other token silently no-ops until its rows arrive. It behaves correctly, but a coupling this load-bearing shouldn't rest on two files agreeing by convention. I'd define the brand tokens once and share them between importer and parser, so they can't drift apart unnoticed.

- **The two lookup paths that drifted should have been one.** The divergence review caught — the hot-path resolver quietly dropping a guard the slower path still honored — was only possible because the two do "the same" brand-aware catalog lookup as separate code. Today's brand-free catalog names are the sole reason they still agreed. A single shared helper would have made the drift impossible instead of merely catchable.
