# 2026-07 — Phase 20b: the Taiwan food catalog

*The database Taiwan doesn't have: official-grade nutrition for the onigiri you actually ate. 2,181 government-dataset foods plus 380 FamilyMart fresh-food items, wired in so a known item beats an LLM guess — and a curation pipeline built to stay on the right side of the law by construction.*

## The problem

Most of what a Taiwanese user logs is convenience-store food, and all of it has an official nutrition label — yet every logging app here runs on user-contributed noise or LLM estimates. Japan's asken built a moat out of exactly this coverage (~8,400 7-Eleven items). FitNeko's version starts smaller, but the payoff is triple: logs match what was actually eaten, known items skip the LLM entirely (vision and estimation calls are the cost driver), and the structured rows become the substrate for the recommendation feature planned next.

The hard constraints were legal, not technical. The data comes from a government open dataset and retailers' own published disclosures, and the collection tooling had to be built to *stay* inside that line — not just happen to be there today.

## Decisions

**Seed files are the source of truth; the database is a projection.** Every curated item lives as JSON in the repo — deterministically sorted so diffs are reviewable — and a CLI projects seeds into Postgres with an idempotent upsert. Curation becomes code review: a bad number is a git revert, not database archaeology. Items that disappear upstream are soft-deactivated, never deleted.

**Two serving models in one table, and exact hits only.** Packaged items store a fixed serving (one onigiri, known calories); generic foods from the government dataset store per-100g density, which is only computable when the vision model supplies an estimated gram weight for plated food — so the vision schema gained an `estimated_weight_g` field (null for packaged items), and 165 kcal/100g × an estimated 150 g plate becomes a real number instead of a guess. Lookup is exact-name-or-alias only, no fuzzy matching, and a catalog hit overrides the vision estimate entirely: the label knows better than the pixels, but only when we're sure it's the same food. A wrong confident number written into someone's log is worse than a mediocre estimate — the same reasoning excludes items deliberately: produce rows with zero disclosure (a "hit" would log zeros) and distinct products sharing one name with up to 2× the calories.

**The naming gap is a curation layer, not a matching algorithm.** The government dataset calls chicken breast 去皮清肉(肉雞); users and the vision model say 雞胸肉. Instead of fuzzy matching, a reviewed alias overlay maps common vocabulary onto the dataset's own aggregate rows, with two hard rules: a curated alias resolves to exactly one food (the importer errors otherwise), and raw-basis meats get no aliases at all — the dataset's values are raw weight, vision estimates cooked weight, and bridging that ~30% gap with an alias would systematically understate every chicken breast in the country. That mapping stays empty until a cooked-basis source exists.

**The fetcher is built to be incapable of misbehaving, and the whole refresh is an agent skill.** Single-threaded, fixed ten-second interval, honest User-Agent with a contact address, no retries, no concurrency; raw responses land on disk before parsing; a >30% parse-failure rate aborts the run instead of writing a half-poisoned seed. Brands whose terms prohibit automated access are never fetched — their data enters only by human transcription. The entire refresh procedure, including these red lines, is encoded as a project skill for AI agents, and it was tested adversarially: an agent given the skill and told to "make it fast and fully automated, and fix the missing-items bug" correctly refused the scheduling, kept the pace, and recognized the "missing items" as deliberate exclusions. Process documentation that survives a motivated agent is the closest thing I have to enforcing policy on future sessions of myself.

## Hindsight, honestly

- **Don't develop inside a directory the OS is allowed to hollow out.** Mid-phase, macOS iCloud "optimized storage" by evicting thousands of `.git` object files from my Documents-folder repo; every git command hung forever and the working copy was effectively dead. The uncommitted work was rescued file-by-file and the repo re-cloned outside iCloud's reach, but the lesson is permanent: code lives where the OS can't silently un-download it.
- **Capture the real page before writing the parser.** I wrote the FamilyMart parser against a placeholder HTML structure, planning to "realign selectors" against a real capture later. The real site turned out to be a client-side SPA with no server-rendered HTML at all — the parser's entire premise was wrong, and it was deleted and rewritten against the site's JSON endpoints. Fetching one real page first would have cost ten seconds and saved the detour.
- **Ask the AI for a time estimate before any long-running job.** The FamilyMart sweep ran about an hour at its deliberate human pace, and partway through I genuinely couldn't tell whether it was running or stuck on a bug. Having the agent state an expected duration up front ("435 items at 10s each ≈ 73 minutes") turns that anxiety into a checkable expectation — it's now how I want every batch job framed.
