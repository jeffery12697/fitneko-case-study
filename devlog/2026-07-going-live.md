# 2026-07 — Going live: CI, a blocked front door, and a same-day teardown

*One day of delivery engineering: the first CI pipeline, an org policy that quietly made the webhook unreachable, a disposable dev environment, tested backups — and then, with the bot verified live on a real phone, a deliberate `terraform destroy`.*

## The problem

Twenty-one phases of features had accumulated with a strong local test story — unit tests, a deterministic mock-tier e2e suite, a LocalStack tier — and zero automation around it. Every merge to `main` was guarded only by whether I remembered to run `make e2e`. Meanwhile the deploy runbook worked, but the stack had never carried real LINE traffic: the webhook's Lambda Function URL was unreachable from the public internet, blocked by an organization-level Service Control Policy — a limitation known since 17d and consciously deferred, since nothing real was wired to LINE yet. Going live meant fixing delivery, not features.

## Decisions

**CI first, and let the mock tier do the heavy lifting.** The e2e suite's mock tier was designed for exactly this: no real credentials, no LLM tokens, deterministic, under a minute. GitHub Actions runs vet, unit tests (with a Postgres service container so the DB-gated tests actually execute instead of skipping), the mock e2e, and a smoke build of every Lambda zip — about three minutes per push, well inside the free tier. The payoff was immediate and embarrassing: the very first CI run on a fresh database exposed a real concurrency bug. Parallel test packages applying migrations to the same empty database collided on `CREATE TABLE IF NOT EXISTS` / `CREATE EXTENSION IF NOT EXISTS` — which are not atomic — and the same race applies to two Lambdas cold-starting simultaneously in production. The fix is a Postgres advisory lock around the migration transaction, plus a regression test that spawns eight concurrent appliers against a scratch database. Locally this had passed for months because the dev database always already existed. A clean-room environment was the test I didn't know I was missing.

**When the platform blocks the front door, change doors — not the handler.** The org SCP denies public (auth-NONE) Lambda Function URLs, so real LINE traffic could never reach the webhook; only IAM-signed synthetic invokes worked. The replacement is an API Gateway HTTP API with a `$default` route: API Gateway invokes the Lambda through service integration, which the SCP doesn't restrict. The handler didn't change by design — a Function URL event *is* the HTTP API payload-2.0 shape — and the swap came with a bonus: the endpoint now survives code deploys, so the URL registered in the LINE console is stable. A false alarm during verification (a curl without `content-type: application/json` got 401 while the same body with it got 200) taught me that HTTP APIs base64-encode bodies they don't recognize as text, which the handler ignored; that became a small hardening fix honoring `isBase64Encoded`.

**A production you can't experiment on needs a sandbox you can destroy.** The moment real users existed, the old apply→verify→destroy habit became a data-loss event. The fix was cheap because the Terraform was already parameterized: one workspace per environment, resource names derived from a `project` variable, and the single hardcoded seam — the Go config's secret names — became a `SECRET_PREFIX` env var. The dev stack applies in minutes, has its own LINE channel and Neon project, and is destroyed freely. The acceptance test for the whole arrangement was destroying dev and watching prod not notice.

**Backups that survive the thing they protect against, with a restore path that can't rot.** `pg_dump` doesn't exist on the `provided.al2023` runtime, and shipping a third-party static binary into a backup path smelled wrong. Instead a small Go Lambda streams every table out via `COPY … TO STDOUT (CSV)` nightly; the schema isn't dumped at all because it's reproducible from the migrations directory — schema-from-migrations plus CSVs is a full restore. Two structural choices matter more than the format: the backup bucket lives in the never-destroyed bootstrap Terraform root, so no workspace's `terraform destroy` can take the backups with it; and the restore sequence is a unit test — dump a seeded database, restore into a freshly-migrated one, compare row for row — so it runs on every CI push instead of decaying in a runbook. Notably, `prevent_destroy` was considered and rejected: it can't vary by workspace, and it would have bricked the disposable dev environment.

**Live, verified, then deliberately torn down.** The bot went live end-to-end — API Gateway to Lambda to Neon to SQS to the worker to a real reply on a real phone. Then I destroyed it the same day, on purpose: deploys are still laptop-driven (`terraform apply` plus an SSO session), and I'd rather bring production up permanently once a proper CD pipeline can deploy it in one auditable click. Idle cost was near zero, so this is a workflow decision, not a financial one. The final data dump is in the bootstrap bucket, and the relaunch checklist — including the trap where secret-name recovery windows block re-creation — is in the runbook.

## Hindsight, honestly

- **Twenty-one phases before CI was a defensible solo-developer trade — but production flips it.** With one developer and no shared mainline, disciplined local verification (`make e2e` before every push) genuinely covered the risk, and I don't regret spending those phases on product instead of pipeline. What changed the calculus is production: the moment real users and a live environment exist, "I remembered to run the tests" is not a quality gate. That the very first CI run caught a real race is the proof — local-only verification had a clean-room blind spot no amount of discipline would have found.
- **The SCP block wasn't a discovery — it was an IOU I wrote in 17d and paid at go-live.** I hit the Function URL restriction during 17d, and since nothing was wired to real LINE yet, I verified the handler with IAM-signed synthetic invokes and deliberately deferred the public entry point. The triage was right; parking a known blocker as one line in a runbook was the weak part. It stayed invisible until going live forced the payment. Known blockers on the critical path deserve a louder home than a parenthetical.
- **The same-day teardown is sequencing, not lost momentum.** The product isn't user-ready yet — the honest bar for running production long-term is the LIFF web view being done, so the bot is something I'd hand to a stranger. Until then, a standing prod is ops attention without user value. The plan is to build CD next, finish LIFF, and then bring production up for good — with the relaunch checklist, the backups, and a one-click deploy making that a boring event instead of a big one.
