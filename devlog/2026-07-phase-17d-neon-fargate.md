# 2026-07 — Phase 17d: optimizing away from the textbook AWS answer

*Building the by-the-book Lambda + RDS serverless stack first — then deliberately dismantling it to a cheaper, simpler one, with an end-to-end suite guarding the move.*

## The problem

The serverless migration was designed as two deliberate movements. The first (phases 17b–17c) stood up the canonical AWS answer: Lambda for both the webhook and the worker, RDS Postgres in a private subnet, an RDS Proxy for connection pooling, IAM database auth, a NAT gateway for egress — the configuration an interviewer expects when they see "Lambda + RDS." That stack works, and building it taught the VPC/NAT/Proxy/IAM-auth material that skipping straight to the cheap version would have skipped.

But it costs ~$45–70/month in load-independent fixed charges — NAT gateway and RDS Proxy bill 24/7 — while FitNeko is a meal-time-bursty, mostly-0-QPS app. Phase 17d is the second movement: migrate the database to Neon (managed Postgres with a built-in pooler and scale-to-zero) and move the worker to Fargate, deleting the VPC tax entirely. The point of doing it in two steps is the judgment it demonstrates — knowing the standard pattern *and* choosing to deviate — not timidity about the target.

## Decisions

**Keep SQS; the Fargate worker long-polls it rather than polling the database.** Two worker trigger models already existed in the codebase: the monolith polls a jobs table (`SELECT … FOR UPDATE SKIP LOCKED`), and the serverless worker consumes SQS. It was tempting to drop SQS and let Fargate poll the table — one fewer component. But continuous DB polling would keep Neon's compute awake around the clock, defeating scale-to-zero — the very thing that justifies moving to Neon. Long-polling SQS touches the database zero times while idle, so Neon can sleep; and it preserves the dead-letter-queue and retry semantics already built and tested in the canonical stack. The "fewer components" optimization would have quietly paid back the savings it was supposed to create.

**One SQS consumer, shared by the Fargate worker and the test harness.** The receive → process → delete loop was extracted into a single component with two entry points: a blocking long-poll loop for the long-running Fargate process (per-message errors are logged and left on the queue for redelivery/DLQ), and a drain-until-empty mode for the end-to-end harness (fail-fast, so a bug surfaces immediately instead of being silently retried). Because both the production worker and the test suite run the *same* consumer code, the e2e run exercises the exact path that ships — the two can't drift.

**Neon deletes the VPC, not just the database bill.** RDS lived in a private subnet, which forced a NAT gateway for outbound calls and an RDS Proxy to survive serverless connection storms — three paid, always-on resources existing only to host and reach a database inside a VPC. Neon is a public TLS endpoint with a built-in pooler, so the worker and webhook reach it directly. The webhook Lambda leaves the VPC entirely (faster cold starts, no ENI), the worker runs in a public subnet with egress straight through the internet gateway, and NAT / Proxy / private subnets are removed. The database swap was the cheap part; eliminating everything the database's location used to require was the win.

**Verify on real infrastructure, then tear it down.** The stack was deployed to a real AWS account and a real Neon project, smoke-tested for the full round trip — a signature-validated webhook call enqueues a job, the Fargate worker pulls it from the queue and reads and writes the database — then destroyed, leaving nothing billable. This is the round trip the private-RDS canonical stack could never complete from a laptop, because its database was unreachable outside the VPC; a public database makes end-to-end verification from anywhere trivial. Deploying for real also surfaced constraints no plan predicts — a Lambda Function URL needs an explicit public-invoke permission even with auth disabled, and free-tier managed databases carry region and retention limits — each caught and fixed against live APIs rather than discovered in production.

## Hindsight, honestly

- **Was building the canonical stack first worth it? For learning, yes; for shipping, no.** The two-movement plan cost real extra time to stand up VPC/NAT/RDS-Proxy/IAM-auth machinery I then deleted. I'd defend it as a deliberate learning exercise — I now understand the textbook answer well enough to argue against it — but I shouldn't pretend it was the fastest path to a running product. If the only goal had been to ship, starting at the Neon target would have been correct.
- **The cross-region reality is a test-only convenience.** AWS compute is in Tokyo; the free-tier Neon region nearest available was Singapore. Fine for a stack that gets torn down minutes after it's verified, but a real launch serving Taiwan/Japan users would want the database co-located with compute — so the "it just works over the public internet" convenience hides a latency decision I've deferred, not made.
- **Fargate is a 24/7 cost too, and I mostly chose it to learn ECS.** For this bursty, mostly-idle load, a worker left on Lambda-behind-SQS might well be the cheaper *and* simpler end state — no always-on task at all. I moved the worker to Fargate as much to have built and reasoned about ECS as because the load profile demanded it. That's an honest bias worth naming: part of this phase optimized for my résumé, not strictly for the bill.
